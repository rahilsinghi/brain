# Fix: Vercel Deploy Preparation for Next.js 16 Build Errors in Karen Project

This commit addresses several frontend build errors encountered when preparing the `karen` project for Vercel deployment, specifically targeting compatibility with Next.js 16 and React 19. Key changes include moving the favicon, refactoring the `useKarenAudio` hook, fixing JSX compliance, and configuring ESLint to bypass false positives during the build process.

## Key Concepts

Vercel Deployment,Next.js 16,React 19,ESLint Configuration,Frontend Build Process,Favicon Management,JSX Compliance,Custom React Hooks

## Details

This commit, `bb130a4`, by Rahil Singhi (co-authored by Claude Opus 4.6), focuses on resolving build errors to ensure a smooth deployment of the `karen` project to Vercel, particularly in the context of `Next.js 16` and `React 19` updates. The specific changes implemented are:

*   **Favicon Relocation**: The `favicon.ico` file was moved from the `app/` directory to the `public/` directory. This is a critical adjustment as Next.js 16 now interprets `app/favicon` as a route, which can lead to conflicts or incorrect asset serving.
*   **`useKarenAudio` Refactor**: The `useKarenAudio` custom hook was updated to fix a recursive reference pattern. This refactor was necessary to ensure compatibility with `React 19`'s strict mode, which often surfaces issues related to `setState` calls within effects or improper hook dependencies.
*   **JSX Compliance in OnboardingFlow**: Quotes within the `OnboardingFlow` component were escaped to resolve `JSX` compliance issues, preventing parsing errors during the build.
*   **Temporary ESLint/TypeScript Disable**: ESLint and TypeScript checks were temporarily disabled during the build process. This decision was made to circumvent false positives generated by `React 19` regarding `setState` calls within effect hooks, which were hindering the build.
*   **ESLint Rule Overrides**: To provide a more targeted solution for the ESLint issues, specific rule overrides were added for `react-hooks/set-state-in-effect` and rules related to `refs`. This allows for more granular control over linting behavior and addresses the `React 19` false positives without completely disabling checks.
