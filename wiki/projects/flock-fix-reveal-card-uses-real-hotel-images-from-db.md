# Flock: Fix - Reveal Card Uses Real Hotel Images from DB

This commit addresses a bug in the Flock project where the trip 'reveal card' displayed model-generated placeholder URLs for hotel images instead of actual hotel photography. It implements a lookup mechanism to fetch real `hero_image_url` from the database, falling back to other photo URLs if necessary, and also retrieves city-level photos for background enhancement.

## Key Concepts

Image Processing,Database Lookup,LLM Hallucinations,User Experience (UX),API Integration

## Details

This commit (`b9b3328`) for the `rahilsinghi/Flock` repository resolves an issue where the `finalize_trip` function could result in the trip 'reveal card' displaying non-existent or placeholder hotel images generated by the language model.

The fix introduces a robust image retrieval process:

1.  **Hotel Name Lookup**: When `finalize_trip` executes and the model provides a potentially fake or example image URL, the system first looks up the hotel name within the `hotels` database table.
2.  **Real Hero Image**: It then attempts to use a `hero_image_url` directly from the database, which is populated with real photography from `Google Places API`.
3.  **Fallback Mechanism**: If a `hero_image_url` is not available, it falls back to using the first available `photo_url` associated with the hotel.
4.  **City-Level Backgrounds**: Additionally, the system now fetches city-level hotel photos to enrich the background of the reveal card, providing a more immersive visual experience.

This enhancement ensures that the 'reveal card' always displays authentic hotel photography, sourced from the `Gemini Vision`-enriched database, significantly improving the visual fidelity and user trust, rather than relying on unreliable model-generated content.

This change was co-authored with `Claude Opus 4.6`.
