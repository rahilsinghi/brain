# Telegram Relay Bot - Always-On Message Queue (brain-relay)

The Telegram Relay Bot, deployed on Fly.io as 'brain-relay', provides an always-on message queue for the `brain daemon`. It ensures messages are received, acknowledged, and queued even when the daemon is offline, processing them within 15 seconds of the daemon coming back online. This architecture leverages webhooks, SQLite for persistent storage, and a REST API for communication between the relay and the daemon.

## Details

The Telegram Relay Bot (`brain-relay`) is a live system designed to ensure continuous message reception and queuing for the `brain daemon`, even when the daemon's host machine (laptop) is offline or sleeping. It was deployed on Fly.io as of 2026-04-16, utilizing a 1GB persistent volume in the ewr region.

## What Shipped
This system provides an always-on Telegram relay, ensuring that messages are always received, acknowledged, and queued. Upon the `brain daemon` resuming operation, all accumulated messages are processed within 15 seconds.

### Architecture
```
User (Telegram) → Fly.io Relay (webhook) → SQLite queue + file storage
                                          ↕ REST API (Bearer auth)
                  Daemon (laptop) polls every 15s → processes → posts response
                  Relay sends response back to user via Telegram
```

### Relay (Fly.io)
The Fly.io component is a `grammY` webhook bot that operates 24/7. Key features include:
-   **Persistent Storage**: Utilizes `SQLite` on a persistent volume to survive Fly.io restarts.
-   **File Handling**: Downloads voice/audio files immediately to prevent Telegram URL expiration.
-   **Message Classification**: Categorizes incoming messages into types like ingest, query, voice, audio, timesheet_entry, timesheet_command, and timesheet_reply.
-   **Instant Acknowledgement**: Provides immediate feedback to the user (e.g., "⏳ Queued", "🔍 Queued").
-   **Scheduled Delivery**: A 30-second loop handles sending reports at configured times.
-   **Response Relay**: A 10-second loop sends responses generated by the daemon back to the user.
-   **[[REST API]]**: Offers a Bearer authenticated API for the daemon to pull/push data.
-   **File Cleanup**: A cron job enforces a 7-day retention policy for files.

### Daemon Integration
The `brain daemon` integrates with the relay through several key components:
-   `src/relay/client.ts`: An [[HTTP client]] for interacting with the relay's [[REST API]].
-   `src/relay/sync.ts`: Manages the poll loop, fetching pending messages, processing them based on type, and posting responses. This module executes real handlers for commands like `/help`, `/start`, `/ts`, `/eod`, `/finalize`, `/invoice`, ingest, query, and voice. Importantly, [[Timesheet commands]] run actual logic here, not generic acknowledgements.
-   `src/index.ts`: When in relay mode, this disables the local Telegram worker and initiates a 15-second `setInterval` polling loop.
-   **Webhook Management**: The webhook is re-registered on every daemon boot to prevent `grammY`'s polling mode from unintentionally clearing it.
-   **Pre-staging**: Daily reports and finalization summaries can be pre-staged to the relay with `deliver_at` timestamps.

### Test Coverage
Extensive testing was conducted:
-   142 relay tests (covering database, webhook, API, scheduler, responder, edge cases, and integration).
-   23 daemon relay tests (client, sync).
-   **165 total new tests**.

### Key Fix: Webhook Clearing
A critical fix addressed `grammY`'s `bot.start()` (polling mode) calling `deleteWebhook` on startup. This previously caused issues when switching from local to relay mode, as the old worker process would clear the webhook. The fix involves re-registering the webhook on every daemon boot when the relay is enabled.

### Key Fix: Generic Command Responses
Initially, the sync loop returned a generic "Command received" for all timesheet commands. This was fixed to execute the actual handler logic, allowing commands like `/help` to return full help text and `/ts` to return actual timesheet data.

### Configuration
The relay functionality is configured via the `.brain/config.yaml` file:
```yaml
# .brain/config.yaml
relay:
  enabled: true
  url: https://brain-relay.fly.dev
  secret: ${RELAY_SECRET}
  poll_interval_seconds: 15
```

### Fly.io Secrets
The following secrets are critical for the Fly.io deployment:
-   `TELEGRAM_BOT_TOKEN`: The same token used for the local bot.
-   `RELAY_SECRET`: A shared secret for authentication between the daemon and the relay.
-   `ALLOWED_USER_IDS`: A whitelist of Telegram user IDs permitted to interact with the bot.
