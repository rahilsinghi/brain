# Vote Keying Fix and Server-Sourced Countdown Implementation (Coro)

This commit for the Coro project addresses a critical issue where votes were improperly keyed by `user_id`, allowing multiple votes from a single user across different browser tabs. It implements a fix by keying votes to a unique `connection_id` per WebSocket. Additionally, it introduces a server-sourced countdown UI for 'drop' events, providing real-time visual feedback to users.

## Details

This commit introduces two key improvements to the Coro project:

### Vote Keying Fix

*   **Problem:** Previously, votes were keyed using a `user_id`. Since `user_id` is often stored in `localStorage` and shared across multiple browser tabs for the same user, this design allowed a single user to submit multiple 'drop' votes from different tabs, bypassing the intended one-vote-per-user restriction for a given session.
*   **Solution:** The vote keying mechanism has been updated to use a `connection_id`. This `connection_id` is a `uuid4` generated by the backend specifically for each accepted WebSocket connection. By linking votes to this unique `connection_id`, the system now correctly enforces a 'one vote per physical connection' policy, preventing users from voting multiple times from different tabs.

### Server-Sourced Drop Countdown

*   **Trigger:** When a total of 3 'drop' votes are accumulated, the backend initiates a countdown process.
*   **Backend Broadcast (`drop_incoming`):** The backend broadcasts a `drop_incoming` message to all connected clients, including a `in_seconds` field (e.g., `{in_seconds: 3}`) indicating the duration of the countdown.
*   **Frontend Action (Countdown Display):** The frontend receives the `drop_incoming` message, reads the `in_seconds` value (rather than relying on hardcoded delays), and immediately initiates a large, prominent countdown timer on the UI. Simultaneously, the relevant UI button or element begins to pulse red, visually signaling the imminent 'drop' event.
*   **Backend Broadcast (`drop_triggered`):** After the specified delay (e.g., 3 seconds), the backend sends a `drop_triggered` message.
*   **Frontend Action (Event Firing):** Upon receiving `drop_triggered`, the 'Lyria update' (presumably the actual 'drop' event or a subsequent state change) is executed, finalizing the drop action.
